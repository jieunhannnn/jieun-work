# 운영 썸네일 자동화 — AI 활용 테스트

## 요구사항

드라이브 폴더에 누끼 딴 상품 이미지를 올리면 피그마에서 보상 유형에 맞는 썸네일이 자동으로 생성·교체·삭제되는 운영 자동화

---

## 문제 정의

- 보상 썸네일(스타상/개근상/우수상 3종) 제작 시 상품 이미지를 매번 수동으로 피그마에 삽입해야 하는 반복 작업 존재
- 보상 상품이 자주 변경되는데, 바뀔 때마다 디자이너가 직접 이미지를 교체해야 하는 리소스 낭비 발생

---

## AI 활용 테스트

**사용한 툴:** Claude Code + Figma MCP + Google Apps Script

구글 드라이브 폴더 구조와 Figma 플러그인을 연동하여 상품 이미지 업로드만으로 보상 썸네일이 자동 생성되는 파이프라인 구축. Google Apps Script를 중간 서버로 활용해 드라이브-Figma 간 동기화를 구현하고, 썸네일 생성/업데이트/삭제 및 순서 정렬까지 자동화함.

---

## 기술 선택 배경

### 왜 Google Apps Script를 썼나?
- 이미지 저장소로 **구글 드라이브**를 선택했기 때문
- Figma 플러그인은 보안 정책상 외부 서비스(드라이브 API)에 직접 접근 불가 → 드라이브 API를 대신 호출해줄 **중간 서버**가 필요
- Apps Script는 드라이브와 같은 구글 생태계라 별도 인증·서버 비용 없이 무료로 웹 API를 만들 수 있어 최적
- 만약 드롭박스를 썼다면 Apps Script 대신 별도 서버(또는 다른 중계 방식)가 필요했을 것

### 왜 피그마 플러그인을 써야만 했나?
- Figma **REST API는 읽기 전용** → 컴포넌트 생성/수정/삭제 같은 쓰기 작업이 불가능
- 피그마 파일에 쓰기를 하는 유일한 방법은 **Plugin API**뿐
- 단, 플러그인은 피그마 앱이 열려 있고 플러그인이 실행 중일 때만 동작
- 이것이 "완전 자동화 불가, 동기화 버튼 클릭 1회 필요"라는 타협의 근본 원인

### 의사결정 흐름 요약
```
드라이브를 저장소로 선택 → 플러그인이 드라이브에 직접 접근 불가 → Apps Script 중계 서버 도입
피그마에 쓰기 필요 → REST API는 읽기 전용 → Plugin API 선택 → 버튼 클릭 1회 타협
```

## 작업 과정

### 1단계 — 구조 설계
- 자동화 목표 정의: 드라이브 업로드 → 누끼 처리 → Figma 썸네일 생성
- 배경 제거 방식 논의: remove.bg API vs Figma 내장 기능 vs 포토샵 Droplet
- Figma 내장 배경 제거는 API로 자동화 불가 확인 → 포토샵 Droplet 방식으로 결정
- 이미지를 미리 누끼 딴 PNG로 올리는 방식으로 간소화 (누끼 유무 자동 감지 기능도 논의)
- 상품 이미지 관리: 구글 드라이브 폴더 구조로 결정

### 2단계 — 네이밍 규칙 확정
- 썸네일 5종에 맞춰 드라이브 폴더 구조 설계
  - `star`, `attendance`, `excellence-diary`, `excellence-gv`, `excellence-live`
- Figma 레이어 네이밍 규칙 확정: `Property 1=폴더-파일명` 형식
  - 예: `star/에어팟.png` → `Property 1=star-에어팟`
- 폴더명이 수상 유형 접두사 역할 → 파일명에는 상품명만 작성

### 3단계 — Figma 파일 구조 분석
- Figma MCP로 TEST 파일 구조 분석
- 컴포넌트 세트 확인:
  - `Product` (26:7776): 상품 이미지 variant 모음
  - `Reward` (26:7607): 5종 썸네일 디자인 (스타상/개근상/우수상 3종)
- Reward 컴포넌트 안에 Product 컴포넌트 인스턴스가 중첩된 구조 파악
- 자동화 방향 결정: Product variant 신규 생성 + Reward 썸네일 복제 후 이미지 교체

### 4단계 — 구글 드라이브 폴더 생성
- 상위 폴더 및 5개 하위 폴더(`star` / `attendance` / `excellence-diary` / `excellence-gv` / `excellence-live`) 직접 생성
- 상위 폴더 ID 확보 → 플러그인 UI에 입력 (ID는 로컬 설정으로 보관)

### 5단계 — Google Apps Script 배포
- Figma 플러그인이 구글 드라이브에 직접 접근 불가 → Apps Script를 중간 서버로 활용
- `appsscript.gs` 작성: `doGet`에서 `action=list`(드라이브 폴더 파일 목록 반환) / `action=image`(이미지 base64 반환) 엔드포인트 구현
- 웹 앱으로 배포 (액세스: 모든 사용자)
- 초기 조직 내부용 URL 문제 발생 → 퍼블릭 URL로 재배포하여 해결

### 6단계 — Figma 플러그인 제작
- `code.js`, `ui.html`, `manifest.json` 3개 파일로 플러그인 구성
- UI: Apps Script URL + 드라이브 폴더 ID 입력, 동기화 버튼, 로그 창
- 주요 기능 구현:
  - 드라이브 파일 목록 스캔 → 신규/기존 판별
  - 신규: Product 컴포넌트 variant 생성 + Reward 썸네일 복제 후 Generated 프레임에 배치
  - 기존: 이미지만 업데이트
  - 삭제: 드라이브에 없는 항목 자동 제거
  - 정렬: 스타상→개근상→우수상 순 자동 정렬
- 설정값(URL, 폴더 ID) 로컬 저장 기능 추가

### 7단계 — 트러블슈팅
- **문제 1**: `manifest.json`의 `networkAccess.allowedDomains`에 Apps Script 도메인(`script.google.com` 등) 누락 → 추가하여 해결
- **문제 2**: Apps Script URL이 조직 내부용으로 배포됨 → 퍼블릭으로 재배포
- **문제 3**: 이미지가 배경에 덮어씌워짐 → Reward 컴포넌트 안의 Product 인스턴스만 타겟하는 방식으로 수정
- **문제 4**: Product 인스턴스 탐색 실패 → `findAllWithCriteria` + 이름 기반 탐색으로 해결
- **문제 5**: 썸네일 정렬 순서 오류 (오토레이아웃 환경) → children 순서 기반 정렬로 수정
- **문제 6**: Generated 섹션을 오토레이아웃 프레임으로 자동 생성하도록 변경

### 8단계 — 추가 기능
- 드라이브에서 삭제된 이미지 감지 시 Figma 컴포넌트 + 썸네일 자동 삭제
- Product 컴포넌트 세트도 Generated 순서와 동일하게 자동 정렬

## 결과

### 기대 결과
- 드라이브에 이미지를 업로드하는 것만으로 피그마 썸네일이 완전 자동으로 생성될 것으로 기대
- 별도의 플러그인 실행이나 동기화 버튼 클릭 없이 백그라운드에서 자동 처리되길 원했으나, Figma REST API가 읽기 전용이라 플러그인이 실행 중인 상태에서만 쓰기 작업이 가능하다는 기술적 제약 존재
- 결과적으로 동기화 버튼 클릭 1회는 필요한 상태로 타협
- 같은 파일명으로 이미지를 덮어쓰면 피그마 썸네일도 자동으로 교체되고, 드라이브에서 이미지를 삭제하면 해당 썸네일도 함께 삭제될 것으로 기대

### 실제 결과
- 플러그인 동기화 버튼 클릭 시 드라이브 폴더를 스캔하여 신규/변경 이미지 자동 감지
- 각 폴더에 따라 해당 수상 유형의 디자인(배경색, 뱃지 텍스트)이 자동으로 매칭되어 썸네일 생성
- 동일 파일명으로 이미지 교체 시 기존 썸네일 이미지 자동 업데이트
- 드라이브에서 이미지 삭제 후 동기화 시 해당 썸네일도 자동 삭제
- 생성된 썸네일은 스타상→개근상→우수상 순으로 자동 정렬되어 배치

### 한 줄평
초기 셋팅은 다소 복잡하나, 상품 이미지가 자주 교체되지 않는 운영 썸네일 프로세스로 정착시키기에 적합함

## 최종 목표

> 드라이브에 상품 이미지만 올리면 → 피그마 썸네일이 자동 완성된다
> **디자이너 도움 없이, 비(非)디자이너군도 언제든 썸네일을 만들 수 있다**
> (이미지 업로드 외 수작업 0 — 배경 제거, 플러그인 버튼 클릭까지 모두 자동)

### 현재 상태 vs 최종 꼴

| 단계 | 현재 | 최종 |
|------|------|------|
| 이미지 업로드 | 드라이브에 수동 업로드 | 동일 (이미 OK) |
| 배경 제거 | 수동 (피그마 기능 직접 사용) | **자동** |
| 썸네일 생성 | 플러그인 버튼 클릭 필요 | **자동 트리거** |
| 제품명 텍스트 | 하드코딩 "제품명 제품명" | **파일명 기반 자동 입력** |
| 이미지 위치/크기 | 불안정 (가끔 배경에 들어감) | **안정적 고정** |
| 결과물 확인 | 피그마 직접 열어봐야 함 | 피그마 열면 바로 보임 (허용 범위) |

### 최종 꼴에 필요한 작업

**① 배경 자동 제거 (누끼)**
- 포토샵을 거쳐서 테스트해 볼 예정 — 포토샵 Action + Droplet으로 원본 이미지를 넣으면 자동 누끼 처리
- Droplet 작업 시 캔버스 크기(265×226px)도 함께 규격화하여 이미지 위치/크기 문제도 같이 해결
- 대안: remove.bg API 연동 (무료 50장/월, 포토샵 테스트 결과에 따라 검토)
- 난이도: 중

**② 동기화 자동 트리거 (버튼 클릭 제거)**
- 근본 문제: Figma Plugin API는 피그마 앱이 열려있고 플러그인이 실행 중일 때만 작동
- 완전 자동화는 Figma가 백그라운드 실행을 지원하지 않아 현재 기술적으로 불가
- 차선책: 드라이브 업로드 감지 시 Apps Script가 Slack/이메일로 "동기화 필요" 알림 발송

**③ 제품명 텍스트 자동화**
- 드라이브 파일명 → 썸네일 내 텍스트 노드 자동 입력 (ex. `star-에어팟.png` → "에어팟" 자동 삽입)
- 난이도: 하

**④ 이미지 위치/크기 안정화**
- 포토샵 Droplet으로 캔버스 265×226px 표준화 후 업로드
- 난이도: 중

**⑤ 이미지 적용 버그 완전 해결**
- 가끔 배경 노드에 이미지가 들어가는 문제 → Product 인스턴스를 ID 기반으로 고정
- 난이도: 중

### 우선순위

```
1순위 (지금 가능) → ③ 제품명 텍스트 + ⑤ 버그 수정
2순위 (단기)     → ① 배경 자동 제거 (포토샵 Droplet 테스트)
3순위 (현실적)   → ② 자동 트리거 대신 Slack 알림으로 대체
4순위 (선택)     → ④ 이미지 크기 표준화
```

이 5가지가 완성되면 디자이너 개입 없이 비디자이너군 누구나 드라이브 업로드만으로 운영 썸네일을 만들 수 있다.
